## The arc
1. **v6.4.11 shipped** — movable Offset & anchor panel (localhost-tested, merged `0ecd8fb`, deployed,
   verified live). Fullscreen-over-preview support: Captain's call — NOT needed. See
   `HANDOVER-AnchorFloatPanel-v6.4.11.md`.
2. **Captain raised a recurring bug:** *"After checkin anim plays the QR repositions and is now partly
   off screen. This has been a problem a couple of times."*
3. **Two failed/partial attempts on the wrong suspect (the logo):**
   - **v6.4.12** (shipped) gave `.v511-stage-logo` a DEFINITE height (was `max-height`) — the theory was
     the async logo load/swap reflowed the stacked logo→title→QR. Captain: *"Logo still changes and
     moves the QR"* then the load-bearing clue: *"Still a no, but Text stayed in place."* (The text
     stabilised; the QR moved INDEPENDENTLY → a QR-specific cause, not logo-stack reflow.)
   - I had deployed v6.4.12 **blind** ("deploy then test") and it failed. Lesson re-learned: don't blind-deploy.
4. **Chrome MCP + localhost sandbox cracked it.** The Chrome extension was connected; a `localhost:8123`
   tab was already open alongside the live tab. I drove the **localhost** tab (no live impact):
   - Forced a plain re-clone → logo + QR stayed put. So the re-clone alone wasn't the bug.
   - Fired the real celebration + sampled rects → DOM dump revealed an `<img>` carrying **two** classes:
     `v511-stage-logo v511-theme-preview-qr`. The LOGO had the QR's identity.
5. **Root cause — `Org.html:9616`, `v511EnsureRuntimeStage`:**
   ```js
   var qr = stage.querySelector('.qr-box, .qr-box-large, .qr-box-small') || stage.querySelector(':scope > canvas, :scope > img');
   ```
   On the inline + fullscreen clones the QR lives in a `.v511-theme-preview-qr` **div** (not a `.qr-box`),
   so the first query missed and the `:scope > img` fallback matched the stage **LOGO** (a direct-child
   `<img>`). The code then did `qr.classList.add('v511-theme-preview-qr')` — tagging the logo as the QR.
   From that one stray class:
   - logo inherited the QR slot's `background:#fff; border-radius` (CSS L4466) → **the "white box"**;
   - QR-motion transforms (`.cel-qr .v511-theme-preview-qr`) hit the **logo** → "grows / changes size";
   - `--qr-cx/cy` (focal point for radial/pivot effects) computed from the **logo** at the top of the
     stage instead of the QR → radial/pivot effects **flung the real QR off-screen**.
   The logo was never guilty — three patches treated a symptom on an innocent element.

## The fix (v6.4.13, commit `c2e5933`)
At `v511EnsureRuntimeStage`: strip the stray class off any logo, then resolve the QR while **excluding**
the logo and preferring the real QR div:
```js
Array.prototype.forEach.call(stage.querySelectorAll('.v511-stage-logo.v511-theme-preview-qr'), l => l.classList.remove('v511-theme-preview-qr'));
var qr = stage.querySelector('.qr-box, .qr-box-large, .qr-box-small')
  || stage.querySelector('.v511-theme-preview-qr:not(.v511-stage-logo)')
  || stage.querySelector(':scope > canvas, :scope > img:not(.v511-stage-logo)');
```
Pill v6.4.12→v6.4.13; SW cache v162→v163. Covers inline + fullscreen (same function serves all stages).

## Methods / traps banked (BOOTSTRAP §6 candidates)
- **Chrome MCP debugging loop:** the extension can eval/inspect/measure the **running localhost tab**
  (and the live tab) directly — `mcp__Claude_in_Chrome__javascript_tool` + `list_connected_browsers` +
  `tabs_context_mcp`. Mechanism-level ground truth without bothering the Captain. (Earlier watches
  thought Chrome MCP "couldn't reach localhost" — that was *navigation*; eval on an already-open tab works.)
- **Freeze-safe resolver test:** `window.fireConfiguredSequence([], 'allow', stage, {theme})` runs
  `v511EnsureRuntimeStage` (the QR/logo resolution we wanted to test) but returns before playing
  anything (empty-sequence guard) — so you verify the fix without the animation.
- **Renderer freeze:** repeated full-celebration fires on localhost spun the renderer (CDP eval timed
  out 45s ×2). Recover with a tab reload. Likely test-harness-specific (rapid re-fires + dev QR render);
  the live site doesn't freeze — but worth knowing if rapid check-ins ever feel sluggish.
- **Internal vs window bindings:** `renderLogoOnStage` / `renderInlineCheckinClone` are called by their
  LOCAL (closure) names inside Org.html even though they're also on `window`; monkey-patching
  `window.X` from the console does NOT intercept the internal call. Test via file-edit + localhost reload.

## Debugging lesson (the real one)
When you've patched the same **visible symptom** twice and it persists, the element you keep patching is
probably **innocent**. Look for the code that mis-identified it — something handed the logo the QR's name
tag. (Fittingly on-brand for a project whose whole job is proving *which* identity is which.)
