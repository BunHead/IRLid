**`v5.11.24` invite QR camera-readability fix.** 4a's camera couldn't decode the 180px M-correction QR carrying ~1500-char invite token. Diagnosis: QR was too dense for the 4a's lower-resolution camera at that physical size on the 8 Pro's screen. Fix:
- Bumped render size 180px → 340px (~3.5× more area)
- Lowered error correction M → L (less redundancy = denser modules but more pixels per module at the larger size)
- Added tap-to-fullscreen handler rendering the QR at 720px on a dark overlay (Captain explicitly requested "double tap to enlarge" pattern; single-tap implemented as cleaner)
- Post-render canvas/img styling forces `max-width: 340px; display: block` on both the canvas (qrcodejs primary) and any img fallback so the rendered output doesn't overflow its container

Bumped pill `v5.11.23a → v5.11.24` and SW cache `v65 → v66`.

**`v5.11.24a` URL wrap + canvas overflow fix.** Captain smoke-tested v5.11.24 on the 4a — camera read the QR, but the content was treated as searchable text (Google search offered), not as a URL. Diagnosis: the QR encoded the bare token payload like `eyJ0eXBlIjoidjVfaW52aXRlIi...` which has no scheme prefix; Android's camera-app URL heuristic requires `http://` or `https://` prefix. Fix: wrap the encoded payload in a full URL before generating the QR:

```javascript
const qrText = window.location.origin + '/Org#staff_invite=' + encodeURIComponent(payload);
```

When 4a scans this, Android offers "Open URL" notification, taps through to `irlid.co.uk/Org#staff_invite=<token>`. Also tightened the post-render canvas overflow fix (still seeing some black-mass bleeding outside QR box in Captain's screenshot from v5.11.24 — applied explicit `display: block; max-width: 340px; height: auto` to both canvas and img post-render). Bumped pill `v5.11.24 → v5.11.24a` and SW cache `v66 → v67`.
