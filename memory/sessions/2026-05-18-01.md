## What we learned

**Architectural commitment banked from Captain's early-watch question.** Captain asked: *"would like to test all functionality of visuals (but you shouldn't need the workers involved if the QR is just a placeholder, correct?)"*. The answer is yes — and this isn't just a mockup convenience, it's a real architectural commitment for IRLid. Visuals (theme, palette, animation, image bg, QR rendering, audio) are all client-side. The Worker is only involved in:
- `POST /org/settings` — saves configured settings to D1
- `GET /org/settings` — loads on next session
- Real check-in flow — Worker mints the nonce that becomes the QR's content string + tracks the attendance state that fires the celebration

This means we can prototype the entire visual surface without backend dependencies. Mockup → spec → implementation in stages. The Worker integration is the last mile, not the first.

**The iterative-tier system works.** Building incrementally — each tier shipping testable improvement — meant Captain could feel changes immediately and direct what mattered next. Tier 1 made things move. Tier 2 made them remember. Tier 2.5 made them controllable. Light mode made them legible in both themes. Tier 2.6 added structural completeness. Tier 3 made the QR real. Tier 3.5 fixed the size + audio. Tier 3.6 closed the colour-palette-flowing loop. Each step was a meaningful Captain-pause-and-test moment, never a 4-hour silent build.

**"Test the visuals without Workers" as a design pattern.** Worth banking generally for future Number Ones: when an UI surface is purely presentational (theme/palette/animation/copy), it can be wired live in the mockup tier without backend. Save the Worker round-trip for the persistence + event-firing layer. This keeps the design iteration loop fast.
