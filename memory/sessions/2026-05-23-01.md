## Notes banked

- **Option C over Option A for Expected list** — Captain chose the architecturally-right path (stable IDs in `org_expected`, events reference by ID) over the quick-win (free-form textarea per event). Port becomes mostly-mechanical mapping rather than redesign-during-port. The "broker, not store" Phase G design for v6.0 (identity-bound prefs via signed envelope) reuses the same pattern.
- **Captain's UX antennae are exceptional.** Spotted: dual scrollbar issue, modal dark-mode regression, room vocab not applying to badges, Now line at 16:00 instead of 13:17, triple scrollbar return, +Add event using today instead of selected day, Invite staff orphan button. Every one a real bug.
- **PowerShell `(function-name)` parser trap** — first hit today (both T4.3.59.5 + T4.3.60 push attempts died). Captain's PowerShell parsed parens in the double-quoted commit message as subexpressions. Fixed by switching to single-quoted commit messages. Banked to BOOTSTRAP §6.
- **`.nojekyll` is mandatory for static-only Pages sites** — the T4.3.59 first-push attempt produced a build failure (#749 red X) because GitHub Pages was running Jekyll on a 1.06MB HTML file. Adding `.nojekyll` (empty file, presence-only) fixed the next deploy. Banked to BOOTSTRAP §6.
- **`isSelectedDayPast` semantics** — viewing past week ALWAYS returns true; viewing future week ALWAYS returns false; current week compares dow against today's dow honouring `weekStartsMonday()`. Used for three +Add event buttons (bottom of tab, list header, swim-lane day-strip header). Consistent disable + relabel pattern.
- **Pages build failure recovery pattern.** When a Pages deploy red-Xs, GitHub continues serving the LAST successful build. So the SITE works (old version), but new commits don't propagate. Diagnostic: click into the failed workflow run; if Jekyll-related, add `.nojekyll`.
