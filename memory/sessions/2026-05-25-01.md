## What I learned (ordered by usefulness)

**Windows case-collapse is the single biggest architectural hazard for this repo on Captain's machine.** When origin has two case-distinct entries (lowercase + capital), Captain's Windows can only hold ONE file on disk. Every git operation that touches a case-related path becomes a potential trap: `git checkout` overwrites the WRONG case; `git add` stages the WRONG entry; `git pull` propagates deletes to WRONG paths. The cure is to make the case-collision impossible by ensuring origin only has one casing per path. The mechanism that worked today: delete the lowercase entry via GitHub web UI (runs on Linux backend, preserves case), pull locally, edit. No more two-cased entries.

**GitHub web UI delete is two clicks, not one.** Trash icon = preview page. Green "Commit changes..." button + popup confirm = actual delete. Captain hit this trap twice today; the third time I labelled "Two clicks. The trash icon is preview; the green button is commit" and that landed.

**`Number(id)` is a type-cast bug for any code that might see UUID-style IDs in the future.** The v5.11 schema reset converted expected_id from integer to UUID-string. Every `Number(id)` in the client became silently broken. There may be other instances I didn't catch — worth a future grep sweep.

**Wrangler tail is the production debug instrument.** The Worker has verbose `console.log` lines on every `requireSignedAction` failure with payload keys, nonce state, type mismatches. When a generic Worker error code is returned, `wrangler tail` shows the actual rejection cause in seconds. Captain used it to confirm the actual cause was `invalid_action_payload` from schema check, not from signature verification or auth.

**The Service Worker's "network-first HTML" strategy doesn't help in-memory JS.** SW network-first applies to navigation requests. Once a tab has loaded Org.html, the JS is in memory until the tab is closed and reopened (or hard-refreshed in a way that re-fetches the HTML). v5.11.0j deploys don't reach the 8 Pro's open tab; Captain needs to close + reopen. Hard refresh via swipe-down on Android Chrome does work; pull-to-refresh on the tab works; just keeping the tab open doesn't.

**OrgCheckin.html and Org.html have separate version histories.** Two different files. Two different pill numbers. They served the same D1 backend in parallel for months. Captain's 8 Pro kept finding its way to OrgCheckin via autocomplete and seeing `v5.11.2` while the desktop was on `/Org` showing `v5.11.0j`. The pill on the URL bar is canonical truth — **always read the URL bar AND the pill together** during dashboard transitions.
