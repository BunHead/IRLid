## The gap
A re-hire is an **already-enrolled device whose membership was removed**, then re-invited on the SAME
phone. It dead-ended: the Captain's Nokia scanned a fresh re-invite and **landed on a stale Check-in**,
the invite silently ignored.

## Trace (three layers, two client + one Worker)
1. **`tryShowSingleDeviceInviteAcceptIfNeeded` gates** (`Org.html`): bailed if `v511InviteAcceptHasSession()`
   (gate 1) OR `irlidV5Enrolled()` (gate 2) — a re-hire is both, so the accept screen never rendered.
2. **Boot ordering** (the part the code-trace alone missed): the session-restore block runs BEFORE the
   invite-accept code. A re-hire still has a session + a cached org snapshot, so `freshOrgFromSession`
   → `/user/orgs` empty (membership gone) → catch → **cached snapshot loaded + `return`** at ~L19679,
   before the invite code at L19704 ever ran. (The code-trace predicted a sign-in dead-end; the Nokia
   showed a stale Check-in instead — **the hardware corrected the diagnosis** and revealed this ordering.)
3. **Worker** (`orgInviteAcceptOnThisDevice`, `irlid-api-org/src/index.js`): hard-rejected a known pub_fp
   with `device_already_enrolled` (a fresh-device-only assumption; the accept path only ever INSERTed a
   brand-new portal_user).

## Fix (v6.4.14, commit `758de24`)
- **Client A** — `captureStaffInviteFromHash` sets `window.__irlidStaffInviteFromHash = true` for an invite
  that came from the URL hash THIS load; a new early call in the boot handler (after DEV_AUTO_LOGIN, before
  the session-restore blocks) runs `tryShowSingleDeviceInviteAcceptIfNeeded()` and returns if it takes over
  — so a fresh invite beats the stale-snapshot restore.
- **Client B** — the accept gate relaxes `v511InviteAcceptHasSession()` / `irlidV5Enrolled()` ONLY when the
  invite is fresh (`__irlidStaffInviteFromHash`). A stale STORED token still can't interrupt a normal
  signed-in user on boot. The accept handler already reuses the existing credential (only enrols if not
  already enrolled), so an enrolled device signs with its existing key.
- **Worker** — re-home a known pub_fp: reuse the existing `portal_users` row (refresh display_name from the
  issuer-signed label), `DELETE` then `INSERT` the membership (lands re-hire AND re-invite-to-change-role),
  issue a session, claim the invite. Every mutating statement guarded on the invite being `pending` (incl.
  the DELETE) so a replay can't delete a membership without re-creating it — claims stay atomic.
