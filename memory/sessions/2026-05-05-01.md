## Architectural design discussions captured this watch

### Lead Admin transfer-of-privilege rule (Captain's design)

**Default state:** an org has exactly **1 Lead Admin**.
**Transfer state (transient):** an org may briefly hold **2 Lead Admins** during handover.
**Forbidden states:** 0 (orphaned), 3+ (committee creep).

**Constraints in code:**
- LeadAdmin can ADD another LeadAdmin iff org has exactly 1 LeadAdmin (transfer state begin)
- LeadAdmin can DELETE a LeadAdmin (incl. self) iff org has exactly 2 LeadAdmins (transfer state end)
- Developer overrides both: can ADD or DELETE in any state (orphan recovery + force-replace deadlocked transfers)

This invariant should be implemented at the Worker layer in Batch C.5 (staff invite scan-in flow) when membership ops first get UI. Worth formalising in PROTOCOL.md §14.9 alongside the existing role permission matrix.

### portal_users vs OAuth users merge — link, don't merge

**Captain's confirmed direction: Option (c) — keep both tables, link them.** Most users will have both: hardware-key for portal admin, OAuth for receipts/recovery. Some have only one. Many-to-many table:

```sql
CREATE TABLE portal_user_external_links (
  portal_user_id TEXT NOT NULL REFERENCES portal_users(id),
  provider       TEXT NOT NULL,         -- 'google' | 'apple' | 'microsoft' | 'github' | ...
  external_id    TEXT NOT NULL,
  linked_at      INTEGER NOT NULL,
  display_label  TEXT,
  PRIMARY KEY (portal_user_id, provider)
);
```

### Three-tier proof hierarchy (Captain's "final proof is the device" push-back)

- **Tier 1 — Hardware-backed credential** (Secure Enclave / TEE / FIDO key). Required for any **write/privileged** action.
- **Tier 2 — OAuth account verification.** Sufficient for **read** access only. NEVER sufficient alone for privileged ops.
- **Tier 3 — Multi-account recovery quorum.** N-of-M linked accounts collectively authorise enrolment of a NEW hardware credential when previous lost.

**Hardware signs, OAuth identifies.** This is the principle.

### User-recovery quorum (Captain's design)

Distinct from the network-level regency Quorum (4-of-7 regents for Captain succession). This is per-user identity recovery.

**Default: 4-of-5 of the user's linked OAuth accounts must independently sign a recovery agreement** before a new hardware credential is enrolled and bound to their existing portal_user. Old pub_fp added to revocation list.

**Captain's confirmed parameters:**
- 4-of-5 default threshold
- Geographic / jurisdictional diversity asked for at link time but not enforced (most providers are US-based; commercial/technical diversity is the practical defence)
- Recovery staged over time — provider signatures collected over different days/sessions to defeat fast-cascade scenarios where an attacker pwns the user's email and chains through multiple OAuth resets in one sitting
- Revocation plumbing required — old pub_fp to revocation list when new credential enrolled

This becomes a clean architectural distinction:

| Layer | Mechanism | Purpose |
|------|-----------|---------|
| Per-user identity recovery | 4-of-5 linked OAuth accounts | User regains access after device loss |
| Network constitutional succession | 4-of-7 standby Quorum + AI-witness ledger | Network continues if Captain incapacitated |

Two quorums, different scopes, both legitimate. Both should be in PROTOCOL.md.
