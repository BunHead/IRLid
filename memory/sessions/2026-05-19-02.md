## Three patches worth banking specifically

**T4.3.7 — Stable-ID rebuild fix.** This one explained the whole class of "buttons don't seem to do anything" reports that had been accumulating across the morning watch. `v511ReadSequenceFromDom` was rebuilding each sequence item fresh on every 300ms debounced save and silently dropping `item.settings`. Every chip selection vanished after 300ms or whenever any field triggered a save. Once stable IDs were assigned per row + matched back via `byId` map, settings preserved across saves. *This bug would have been invisible in a sweeping refactor; the patch-per-feature cadence is what caught it.*

**T4.3.22 — Play with previous toggle.** Captain explicitly banked this earlier ("add to list, play with previous"); shipped it tonight as a small `+` button next to each row's duration. Pressed = step fires concurrently with step above. Architecture extension was a `chainNextNow()` helper in `v511PlaySequence` that bypasses the cleanup setTimeout when next step is marked withPrev. CSS `:first-child` hides the button on row 0 naturally — drag a row to position 0 and the button vanishes; drag it back and it returns. No JS visibility logic needed.

**T4.3.26 — WAV save round-trip.** Bug Captain noticed: uploaded WAVs didn't persist across reload. Diagnosis: `uploadedWavs[mode]` + `uploadedWavNames[mode]` were globals defined outside `modeStates` and never captured by `v511CaptureState`. Fix added them to save state + restored them in `v511LoadState` step 6.9. Plus: upload handler now fires `v511SaveDebounced()` immediately (previously it only updated globals, save fired only if some other field changed). Plus: Reset button restoring synthesised tone fallback. Plus: mode-aware filename label so each mode shows its own (or default). Plus: tiered quota fallback — strip imageData first → WAVs next → minimal save — so the entire save doesn't fail when localStorage chokes on heavy data URLs.
