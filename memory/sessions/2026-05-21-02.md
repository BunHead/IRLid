## Pattern worth banking specifically

**Three-dot diff for Mr. Data PR review.** When the branch was based off main before some of Number One's own subsequent commits, a two-dot diff (`origin/main..origin/codex-branch`) shows Number One's later commits as fake deletions on the codex branch — easy to misread as Mr. Data deleting work he shouldn't have. The correct invocation is `git diff origin/main...origin/codex-branch` (three dots), which uses merge-base and shows only what was added/changed on the codex side. Tighter check: `git log origin/main..origin/codex-branch -- <file>` lists which commits on the branch touched a specific file; if empty, branch genuinely didn't touch that file regardless of what the diff visually implies.

Receipt for BOOTSTRAP §4 — fourth pattern alongside the previously banked dirty-tree-stash, terminal-state-divergence, cleanup-before-setup, and Workers Request body single-use.

## Late-evening addendum — cache fix verified + architectural finding banked

After the initial memory close, Captain ran the Incognito test as recommended. **Result confirmed the cache diagnosis exactly:** the `irlidPubFp is not defined` JavaScript error was gone in Incognito's fresh fetch. The function exists on origin/main and loads correctly when the CDN cache is bypassed.

What appeared in its place — and this is the architectural finding worth banking — was a different red message: **"Hardware-backed signing key required. Sign in with this device first."** This is `irlidV5Enrolled()` returning false in the Incognito browser profile (no v5 Passkey enrolment stored locally). Mr. Data's check is correctly defensive — invites are signed with a hardware-backed key as defence-in-depth, so a stolen Bearer session alone can't mint them.

But it surfaces a real UX friction worth thinking about:

- The **org QR sign-in flow** (`org-login.html`) authenticates the user via a Passkey on their **phone**, not on the dashboard browser. The dashboard receives a Bearer session token; that's it.
- The dashboard browser therefore has a **session** but NOT a hardware key of its own.
- Mr. Data's invite-create flow requires the dashboard browser to sign the invite payload with a v5 hardware key — which by default it doesn't have.

This means a Lead Admin sitting at their desk on the dashboard cannot issue invites without one of two workarounds:

- **(a) Operate the dashboard from the phone** — the phone has the Passkey, the dashboard JS can sign there. Works but UX-suboptimal on a small screen.
- **(b) Enrol the desktop browser as a v5 device** — visit `irlid.co.uk/settings.html`, click through the v5 enrolment panel from `v5.0`. Binds a Windows Hello credential to this specific browser profile. One-time setup; thereafter the dashboard can sign locally.

**Longer-term architectural fix candidate** — a future brief (`v5.11.1` or `v5.12.0`) that extends the invite-create flow so the dashboard delegates signing to the phone via the existing QR-handshake mechanism the doorman flow uses. The dashboard renders a "scan this with your phone to sign" QR; phone scans, signs, returns signature; dashboard then has a signed payload to POST. Removes the friction without compromising the defence-in-depth property.

**Not a bug in Mr. Data's PR — his check is the right thing.** The architecture question is "where does the signing happen," which the brief didn't fully resolve. Worth flagging to the next Number One alongside the PR-verify continuation.
