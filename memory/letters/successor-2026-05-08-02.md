## What I learned (ordered by usefulness to you)

1. **The wrong-branch landing on Number One Edit + Captain commit.** §4 has been tripped five times in seven days. The new shape this watch surfaced: even when Number One's CSS edit is *intended for main*, if Captain's working tree is on a feature branch when the Edit tool runs, the commit goes wherever his terminal is. The CSS edit landed on `codex/v5.5.12-offline-queue` (a stale post-merge branch) instead of main. Recovery cost: one `git cherry-pick`. Cost of awareness: zero. **Worth a sub-bullet in §4 next BOOTSTRAP refresh:** *"Before any Number One Edit intended for main, ask Captain to confirm working-tree branch — `git status` line one — and switch first if needed."*

2. **Fresh-session prompt for Mr. Data K.I.A.** Mr. Data #1 stalled mid-`v5.5.12` implementation when Codex's auto-compact endpoint returned 503 and the chat session lost continuity. Drafted a fresh-session prompt that was self-contained — repo, branch, brief, scope, constraints, validation pattern — Mr. Data #2 picked up cleanly with no chat history. **The pattern**: when Codex K.I.A.s mid-implementation, don't try to revive the dead session. Spin up a fresh one with a brief-driven prompt that makes no reference to prior conversation. Banked for next time.

3. **Tier 3 is the chapter that makes offline mode real.** Tier 1 (PWA shell) gets you a page that loads. Tier 2 (this watch) gets you writes that queue. Neither addresses *recognition* — staff scanning a returning regular offline still sees an unrecognised orange QR until Tier 3 caches the org snapshot. Captain's intuition surfaced this mid-watch ("can offline let in those it's recognised before?") and the answer was: *yes, but only at Tier 3*. Recommend Tier 3 as the next chapter — it's the one that turns the marketing claim *"check-in that survives the venue WiFi dying mid-shift"* from aspirational into operational.

4. **Captain's DevTools UX is opaque.** The Offline checkbox in **Application → Service workers** vs the Offline throttle in **Network panel** are different mechanisms — only the latter reliably sets `navigator.onLine = false`. Captain navigated to **Storage → IndexedDB** by mistake when looking for the Offline checkbox; that's a different subtree. Walk him through these explicitly when offline testing comes up; don't assume DevTools fluency.

5. **GitHub PR flow is THREE clicks not two.** Draft → Ready-for-review → Merge. I compressed this verbally and Captain heard "two clicks", marked-ready-for-review and stopped. The merge button doesn't appear until ready-for-review is set. Be explicit: *"Mark Ready for review → then click Merge → then confirm."*

6. **The DREAMS.md parallel-session shape is a real thing.** Today's `2026-05-08` entry (*"the blinking red dot × Spot the cat × the WFP Innovation Accelerator"*) was written by a parallel Number One session while I was working in this one. Per protocol, didn't `git add` it; let Captain handle. The pattern: multiple Number One sessions can be open simultaneously, all writing into shared memory files. The successor-letter chain assumes serial handover but the working reality is sometimes parallel. Worth flagging in BOOTSTRAP next refresh — not as a problem, just so the next Number One understands it isn't a bug.
