### 1.3 Privacy Mode vs Anonymous Mode — overlap

Help text reads:
- Privacy Mode: *"Prefer reduced personal detail in the entry flow."*
- Anonymous Mode: *"Permit anonymous-style handling where appropriate."*

The OLD-style version (line 769) is more concrete: *"Count only — no names stored."* That's a meaningful distinction from Privacy Mode (which keeps the name but hashes the GPS).

**My read:** the two ARE semantically distinct, but the help text in the new switch-tile UI doesn't make the distinction clear to an admin reading them cold. Recommendation:

- Privacy Mode → "GPS coordinates stored as a SHA-256 hash, not raw lat/lon. Names retained."
- Anonymous Mode → "Count attendance without storing names. Aggregate-only."

Two different privacy postures, two different use cases (a clinic / a counted-attendance event), worth keeping both with sharper labels.

### 1.4 Storage path — URL-param vs `settings_json`

Schema has `organisations.settings_json TEXT NOT NULL DEFAULT '{}'`. **It's not currently being written by the Settings panel** as far as I can see — settings flow into URL parameters of the Check-in QR (logo, welcome, redirect, score, mode). Each org admin's settings are local to whatever URL they last generated; they don't persist server-side beyond that QR.

This works for the current "one event, one QR" pattern but doesn't scale to:
- An org with multiple concurrent events
- Settings that need to apply across all of an org's QRs
- Theme settings (which we're about to add) that should be org-level not event-level

**Recommendation:** start writing `settings_json` server-side as the canonical org-level state, and treat URL params as a per-event override layer on top. Theme settings should be org-level (settings_json) by default.

## 3. Theming panel design

### 3.1 What to add

Per Captain's brief: a Theming panel beneath Branding. Three colour pickers:

- **Primary colour** — main UI accent, button colour, active-state colour.
- **Accent colour** — secondary accent, hover states, badges.
- **QR foreground colour** — colour of the QR modules. **Subject to readability constraints (see §3.3).**

### 3.2 Colour picker UX

**Recommendation: HTML5 `<input type="color">` for all three.** Cross-browser, no library, opens the OS-native colour wheel/picker. Renders consistently on mobile (where most Org admins might be testing). Free.

If you want a fancier picker (gradient/HSV wheel/swatches) we could add `iro.js` as a CDN library, but for v1 the native picker is the pragmatic choice. **Going with native unless Captain wants the upgrade.**

### 3.3 QR readability constraints — critical

**Bare colour pickers on QR foreground are dangerous.** A user picking a light yellow on white background creates an unscannable QR. Three guard-rails:

1. **Lock background to white.** QR background stays `#FFFFFF`. Only the foreground (modules) is user-coloured. This is the convention in QR styling tools and reliably works.

2. **Contrast check before accepting.** Compute relative luminance ratio between chosen foreground and white background. Per WCAG / QR standards, **minimum 4.5:1 contrast**, recommended 7:1 for reliable scanning across cameras. If user picks something below 4.5:1, show a warning ("This may not scan reliably on older cameras") and offer a one-click "use a darker variant" suggestion.

3. **Live QR preview.** As they pick a colour, re-render a sample QR with the chosen modules colour next to the picker. Visual feedback they can scan with their phone right there. Best possible test.

### 3.4 Storage

Theme is org-level, not event-level — it should persist across all Check-in QRs the org generates. Two storage paths:

**Option A — extend `settings_json`** on the `organisations` table (already exists in schema, currently underused).
- Pro: single source of truth, scales beyond URL params.
- Con: requires a new endpoint to write/read it (`PATCH /org/settings`).

**Option B — keep URL params** for theme too (`&primary=23A4FF&accent=FF6B35&qr=1A1A2E`).
- Pro: matches the existing pattern, no Worker changes needed.
- Con: doesn't scale to multiple events, theme has to be re-set every time a new QR is generated.

**My recommendation: A.** This is the right moment to start using `settings_json` (per §1.4 above). The theming panel's three colours go into `settings_json.theme = { primary, accent, qrForeground }`, the Org admin sets them once, all subsequent Check-in QRs inherit. Per-event overrides via URL params remain possible for special cases.

### 3.5 Where the theme applies

Three surfaces consume the theme:

1. **`org-entry.html`** (the public Check-in QR landing page) — uses Primary for the main button colour, Accent for hover/active, QR foreground for the rendered code.
2. **`OrgCheckin.html`** (the Dashboard) — uses Primary for active nav item, key buttons. Could use Accent on stat cards. QR foreground applies to the displayed Check-in QR.
3. **Receipts displayed at the org context** — could pick up Primary/Accent for the org-coloured corner of a verified receipt. Future work.

### 3.8 Implementation — JS

```js
// Theme settings handling — add to OrgCheckin.html script block

const THEME_DEFAULTS = {
  primary: "#3FB950",
  accent:  "#58A6FF",
  qrFg:    "#000000"
};

function applyThemeToPage(theme) {
  const t = Object.assign({}, THEME_DEFAULTS, theme || {});
  document.documentElement.style.setProperty('--theme-primary', t.primary);
  document.documentElement.style.setProperty('--theme-accent',  t.accent);
  // QR foreground is consumed by renderQR helpers, not via CSS vars.
}

// WCAG-ish relative luminance, returns 0..1
function relLuminance(hex) {
  const m = /^#?([0-9a-fA-F]{6})$/.exec(hex);
  if (!m) return 0;
  const v = parseInt(m[1], 16);
  const channels = [(v >> 16) & 255, (v >> 8) & 255, v & 255].map(c => {
    const s = c / 255;
    return s <= 0.03928 ? s / 12.92 : Math.pow((s + 0.055) / 1.055, 2.4);
  });
  return 0.2126*channels[0] + 0.7152*channels[1] + 0.0722*channels[2];
}

function contrastRatio(fg, bg) {
  const l1 = relLuminance(fg), l2 = relLuminance(bg);
  const [a, b] = l1 > l2 ? [l1, l2] : [l2, l1];
  return (a + 0.05) / (b + 0.05);
}

function updateQrContrastHint(fgHex) {
  const ratio = contrastRatio(fgHex, "#FFFFFF");
  const hint = document.getElementById('themeQrContrastHint');
  if (!hint) return;
  if (ratio >= 7) {
    hint.textContent = `Contrast ${ratio.toFixed(1)}:1 — excellent for QR scanning.`;
    hint.className = 'theme-hint';
  } else if (ratio >= 4.5) {
    hint.textContent = `Contrast ${ratio.toFixed(1)}:1 — readable, but darker would scan more reliably on older cameras.`;
    hint.className = 'theme-hint warn';
  } else {
    hint.textContent = `Contrast ${ratio.toFixed(1)}:1 — too low. QR may not scan reliably. Pick a darker colour.`;
    hint.className = 'theme-hint error';
  }
}

function renderThemePreviewQr(fgHex) {
  const box = document.getElementById('themePreviewQrBox');
  if (!box) return;
  box.innerHTML = '';
  // Use the same QRCode lib already loaded in OrgCheckin.html
  if (typeof QRCode !== 'undefined') {
    new QRCode(box, {
      text: 'IRLid theme preview',
      width: 80,
      height: 80,
      colorDark: fgHex,
      colorLight: '#FFFFFF',
      correctLevel: QRCode.CorrectLevel.M
    });
  }
}

function readThemeFromUI() {
  return {
    primary: document.getElementById('themePrimary').value,
    accent:  document.getElementById('themeAccent').value,
    qrFg:    document.getElementById('themeQrFg').value
  };
}

function writeThemeToUI(theme) {
  const t = Object.assign({}, THEME_DEFAULTS, theme || {});
  document.getElementById('themePrimary').value = t.primary;
  document.getElementById('themeAccent').value  = t.accent;
  document.getElementById('themeQrFg').value    = t.qrFg;
  updateQrContrastHint(t.qrFg);
  renderThemePreviewQr(t.qrFg);
  applyThemeToPage(t);
}

function wireThemePanel() {
  const inputs = ['themePrimary', 'themeAccent', 'themeQrFg'];
  inputs.forEach(id => {
    const el = document.getElementById(id);
    if (!el) return;
    el.addEventListener('input', () => {
      const theme = readThemeFromUI();
      portalState.theme = theme;
      applyThemeToPage(theme);
      if (id === 'themeQrFg') {
        updateQrContrastHint(theme.qrFg);
        renderThemePreviewQr(theme.qrFg);
      }
      // Persist (when settings_json wiring lands):
      // queueSettingsSave({ theme });
    });
  });
  const resetBtn = document.getElementById('themeResetBtn');
  if (resetBtn) {
    resetBtn.addEventListener('click', () => {
      portalState.theme = { ...THEME_DEFAULTS };
      writeThemeToUI(portalState.theme);
    });
  }
  // Initial render
  writeThemeToUI(portalState.theme || THEME_DEFAULTS);
}

// Call wireThemePanel() on page load, after portalState is initialised.
```

The theme writes to `portalState.theme = { primary, accent, qrFg }`. Existing `portalState` is the canonical settings holder — adding `theme` extends naturally.

### 3.9 Worker schema/endpoint changes (for full server-side persistence)

If we go with §3.4 Option A (settings_json server-side), the Worker needs:

**New endpoint** `PATCH /org/settings`:
- Auth: existing org API key
- Body: `{ settings: { theme: { primary, accent, qrFg }, ... } }`
- Behaviour: merge into `settings_json`, return updated.

**Worker validation:**
- `primary`, `accent`, `qrFg` must each match `/^#[0-9A-Fa-f]{6}$/`.
- QR contrast check: server-side recompute, reject if `qrFg` contrast against white < 4.5:1. This stops a misconfigured admin from generating unscannable QRs even if they bypass the frontend warning.

**Schema:** no migration needed — `settings_json` already exists.

This is a small Worker change — probably 30-40 lines of code. Could fold into the Batch 6 PR if the bake-off submitter wants to do it together, OR stand alone as Batch 6.5 / Batch 7.
