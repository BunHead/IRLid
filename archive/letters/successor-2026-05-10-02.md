## What I learned (lessons banked, please absorb)

**1. Diagnostic-first beats hypothesis-first when you're past one round of failed assumptions.** Watch-1 had spent significant effort on the "is `LOGIN_DEBUG === '1'` strict-equality failing for opaque reasons?" hypothesis. Tonight's session sidestepped that question entirely with **unconditional `console.log` at every 401 site + always-include-debug in the response body** — no env-flag gates. First retry produced the smoking gun in the response body itself: `debug_bootstrap_fp_len: 1, debug_bootstrap_fp_first4: ""`. Total time from "deploy diagnostic" to "smoking gun": ~5 minutes. The lesson: when LOGIN_DEBUG-gated diagnostics aren't surfacing the answer, sidestep the gate entirely with unconditional logs. Don't waste a watch debating whether the gate is broken.

**2. Mandatory `_len` / `_first4` / `_last4` debug fields for secret-dependent 401s are the canonical detection pattern for the Ctrl+V trap.** Without those fields, the SYN-byte secret looks indistinguishable from a normal fp_mismatch. With them, the answer is unambiguous in one round.

**3. Trust the diagnostic outputs over the inherited successor letter.** Watch-1's letter said `BOOTSTRAP_DEVELOPER_FP=uSwaWJc9r5uSCBbI`. The diagnostic computed Captain's actual fp on irlid.co.uk RP-ID as `TvklFsivZk68R67j`. Always use the diagnostic-derived value; the previous Number One may have a typo or device-mix-up baked into the handoff.

**4. The `irlid_mock_*` localStorage namespace is a test-env-leftover landmine field.** Anywhere the live dashboard falls back to `_mock_` storage, it's reading test-env fixtures. The `_mock_` prefix in OrgCheckin.html should probably never run on `irlid.co.uk` — gate by hostname or remove. Pending-work has the audit task; pick it up early in the next watch so we don't re-encounter shaped variants.

**5. BOOTSTRAP §4 cherry-pick precondition + dirty-tree precondition both held tonight.** No git misadventures. The receipt #8 lead-in pattern (`git cherry-pick --abort 2>$null ; git rebase --abort 2>$null ; git merge --abort 2>$null ; git switch main`) is now muscle memory for Captain. Working as intended.

**6. Worker-source-on-disk vs Worker-deployed-on-Cloudflare are independent — exploit that.** The diagnostic was added to disk + deployed to Cloudflare via wrangler, then reverted from disk + redeployed clean. Git was never touched. Result: zero diagnostic noise in main's history, while the diagnostic was operational on live just long enough to surface the smoking gun. Pattern is replicable for future short-lived diagnostics.

**7. Architectural finding worth banking: BOOTSTRAP_DEVELOPER_FP is a platform-level escape hatch.** Captain's pub_fp matching the secret grants developer access to ANY org on this Worker via `requireDevOrStaffSession` and `requireFreshStaffProof`. Membership rows are needed only for UX surfacing (org switcher). For orgs Captain creates via `/user/create-org`, membership is auto-inserted; for orgs created by other paths (seed, future admin tooling), Captain has Worker-level access but needs manual membership INSERT for UX visibility. This is the right architecture — Worker is the security boundary; client is UX. A future "all orgs" admin view would close the UX gap, but is not urgent (Captain is the only platform admin).
