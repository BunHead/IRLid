Mr. Data + Number One can collaborate cleanly within a single watch.** PR #104 (orange-QR payload survival) was Mr. Data's; on top of his merge I shipped the `device_key_scan` → `doorman_scan` routing fix as Number One inline. Rebase-recovery handled the local-behind-remote case. Net: two chained fixes in one PR-and-commit cycle.

**3. The "test-env file-copy → live" trap is recurring.** Tonight saw FOUR instances:
- Two `OrgCheckin.html` ones (v5.9.0.1 fixed earlier today)
- Two `scan.html` ones (v5.9.0.6 fixed tonight)
The pattern: any string-literal URL pointing at `bunhead.github.io/IRLid-TestEnvironment/...` survives file-copy and silently misroutes. Fix pattern is always the same: `window.location.origin + window.location.pathname.replace(/\/[^\/]*$/, '')`. Now in BOOTSTRAP §6 with sweep guidance.

**4. `git pull --rebase` is the recovery for "local behind remote after GitHub-UI merge".** When Mr. Data's PR is merged via GitHub UI, Captain's local main is suddenly 2 commits behind origin/main. Number One inline edits made between merge and pull land on the OLD parent, push gets rejected as non-fast-forward, recovery is `git pull --rebase origin main ; git push`. Worth banking as a §4 receipt, alongside the existing branch-state and dirty-tree variants.

**5. The "stand for a while" milestone is on production now, not just test env.** Both environments have working e2e check-in/check-out cycles. Live has working bootstrap + sign-in + dashboard + attendee orange-QR + staff scan handoff. The remaining live work (PR #104 forward-port + v5.7.1z.1 forward-port) is incremental polish on a working foundation, not bridge-spanning.
