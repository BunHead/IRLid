## Captain's 30 April design clarifications captured

These are protocol-level decisions, not just UX:

1. **Float cog: removed permanently.** Settings via sidebar nav only.
2. **Initials for Expected list — all rows always shown** (Captain overrode Number One's earlier fuzzy-match suggestion as too complex). Default format `SRA` (compact, screen-reader friendly with `aria-label`).
3. **Event-end grace period: default 1 hour, configurable per-org.**
4. **Deny path writes `event_attendance` row with status `rejected`, then ends.** No redirect to Org website (Accept path does that, Deny intentionally doesn't). Builds the forensic audit trail described in `THREAT-MODEL.md §IX`.
5. **Manual Acceptance: staff with active session can tap-to-log-in attendees, but the receipt carries a "Manual Acceptance" disclaimer.** Lower trust score than cryptographic auto-accept. Distinguishes attended-with-rigorous-proof from attended-with-staff-vouch.
6. **Accessibility is non-negotiable.** Every label needs proper ARIA. Number One must apply this to all UI work.
7. **Co-presence-only for cross-org admin add.** No email/OTP path. Adding a Lead Admin in America requires someone physically there. Captain explicitly accepts this as the protocol's moat. (Per `THREAT-MODEL.md §VIII.3`.)
